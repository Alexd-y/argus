# ISS — Hide "Findings" nav entry for `operator` role

**Issue ID:** ISS-T20-013
**Owner:** Frontend / Admin console + RBAC
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — UX consistency nit (S3-3)
**Priority:** LOW (no security gap; operator without `tenant_id`
already gets a 403 from the backend; visible-but-unusable nav entry
is a UX paper-cut, not a leak)
**Date filed:** 2026-04-21

---

## Context

The admin chrome
[`Frontend/src/app/admin/AdminLayoutClient.tsx`](../../../Frontend/src/app/admin/AdminLayoutClient.tsx)
renders a navigation list including:

* Tenants
* Scans
* LLM
* System
* **Findings** ← added by T20

Each nav item is an unconditional `<Link>`. RBAC enforcement happens
**inside** each route via `<AdminRouteGuard>` and the server-action
RBAC check, so an `operator` clicking "Findings" lands on a
forbidden / empty-state page instead of the data they hoped to see.

The other nav entries follow the same pattern (no role-based
hiding); but `Findings` is the only entry that presents a
super-admin-leaning view (cross-tenant table by default).
`operator` cannot meaningfully use it.

## Problem

* `operator` sees a nav entry that always lands on an empty
  state — minor confusion, repeated on every visit.
* `admin` (without a session-bound tenant) lands on the explicit
  empty state added in S1-6 — same paper-cut.
* No visual hint at the nav level that the destination is
  RBAC-restricted.

## Proposal

1. **Role-aware nav** in `AdminLayoutClient.tsx`:
   ```tsx
   const role = useAdminAuth().role;
   const navItems = useMemo(() => {
     const base = [/* tenants, scans, llm, system */];
     if (role === "operator") return base;            // hide findings
     return [...base, { href: "/admin/findings", label: "Findings" }];
   }, [role]);
   ```
2. **Backend invariant unchanged** — RBAC stays enforced inside the
   route + the server action; nav-level hiding is UX only, not a
   security boundary.
3. **Tests** — Vitest:
   * `operator` role → nav does not contain "Findings" link.
   * `admin` role → nav contains "Findings" link.
   * `super-admin` role → nav contains "Findings" link.
4. **Documentation** — extend
   `ai_docs/develop/components/admin-rbac-matrix.md` (NEW or extend)
   with the nav-visibility table.

## Acceptance criteria

* (a) `operator` does not see the "Findings" nav entry.
* (b) `admin` and `super-admin` see it.
* (c) Direct URL navigation to `/admin/findings` still works for
  `admin` and `super-admin` (nav hiding is not a hard gate).
* (d) Direct URL navigation as `operator` still hits the existing
  empty-state / forbidden message (defence-in-depth unchanged).
* (e) ≥ 3 new Vitest cases for the nav role-matrix.

## Why deferred from T20 fix cycle

* S3-3 grade: UX nit, no security or data-correctness impact.
* The fix cycle was scoped to S0/S1/S2; nav-level role hiding is a
  cross-cutting pattern that should land alongside similar tweaks
  for other RBAC-restricted entries (likely surfaced in T22 audit
  viewer review).

## Dependencies

* `useAdminAuth` already exposes `role` (no new API needed).
* Coordination with T22 / T26 reviewers for consistency on the
  nav-hiding pattern.

## Rationale

Hide-vs-show at the nav level is a sustained UX pattern in admin
consoles; the fix is small and self-contained but not urgent.
Defer until the broader nav-RBAC pattern can land as a single
coherent change covering all role-gated destinations.
