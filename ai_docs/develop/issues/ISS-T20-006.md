# ISS — Decompose `FindingsTable` → `FindingsRow` + `FindingsDrawer`

**Issue ID:** ISS-T20-006
**Owner:** Frontend / Admin console
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — refactor; queue for next cycle
**Priority:** LOW (maintainability; no functional impact)
**Date filed:** 2026-04-21

---

## Context

[`Frontend/src/components/admin/findings/FindingsTable.tsx`](../../../Frontend/src/components/admin/findings/FindingsTable.tsx)
currently houses three concerns in one file:

1. The virtualized table (`@tanstack/react-virtual`) — rows, header,
   sentinel, sort indicators.
2. The detail drawer (overlay, focus trap, per-row export, JSON-tree
   evidence preview).
3. The export popover state (per-row `format` selection,
   `downloadFindingsExport` invocation, error toast).

After the T20 fix cycle the file grew to ~700 lines (focus management
for S2-1 + per-row export for S1-5 + the existing virtualized table).

## Problem

* Single-file size makes each change risky: a tweak to the drawer's
  focus trap forces re-reading 500 lines of unrelated table-virtualisation
  logic.
* Test setup is heavy because every test mounts the full virtualised
  table even when only the drawer behaviour is under test.
* The drawer is a candidate for re-use in `/admin/audit` (audit-log
  detail view) — currently locked inside the table file.

## Proposal

1. **Extract `<FindingsDrawer>`** to
   `Frontend/src/components/admin/findings/FindingsDrawer.tsx`:
   * Props: `item: AdminFindingItem | null`, `onClose: () => void`,
     `effectiveTenantId: string | null`, `triggerElement?:
     HTMLElement | null` (for focus restoration).
   * Owns: backdrop, focus trap, focus restoration, per-row export
     popover.
2. **Extract `<FindingsRow>`** (and `<FindingsHeaderRow>`,
   `<FindingsSortHeader>`) to
   `Frontend/src/components/admin/findings/FindingsRow.tsx`:
   * Memoize via `React.memo` with shallow-prop comparison.
   * Receives the row item and `onSelect(item, triggerEl)` callback.
3. **Slim `<FindingsTable>`** down to ≤ 250 lines: only orchestration
   (virtualizer, row mapping, drawer mount, sentinel for infinite
   scroll).
4. **Tests** — split
   `Frontend/src/components/admin/findings/FindingsTable.test.tsx`
   into three files mirroring the new modules; preserve all 18
   existing assertions.

## Acceptance criteria

* (a) `FindingsTable.tsx`, `FindingsDrawer.tsx`, `FindingsRow.tsx`
  each ≤ 250 LoC.
* (b) Existing 18 Vitest cases pass without behavioural change.
* (c) `<FindingsDrawer>` is consumed by at least one external test
  file directly (proves the API is decoupled).
* (d) No new public exports leak from `Frontend/src/components/admin/findings/`
  beyond the three components and existing types.
* (e) Bundle size unchanged (within ±2 %); tree-shaking still works.

## Why deferred from T20 fix cycle

* The fix cycle's atomicity rule says "one commit per task";
  decomposition would inflate the diff and obscure the security /
  correctness fixes (S0-1, S1-1..6, S2-1..2) the reviewer asked for.
* No functional / security impact; pure code-organisation refactor.

## Dependencies

* None. Pure frontend refactor.

## Rationale

The fix cycle made `FindingsTable` larger (added focus management +
per-row export). Decomposing now keeps each module aligned with a
single responsibility (SOLID — SRP) and unblocks re-using the drawer
on the upcoming audit-log viewer (T22).
