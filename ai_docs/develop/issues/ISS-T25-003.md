# ISS — Extract shared admin-router helpers to `admin_core.py`

**Issue ID:** ISS-T25-003
**Owner:** Backend / Refactor
**Source task:** T25 (Cycle 6 Batch 3 — chain integrity verification API)
**Source review:** post-review fix-cycle for commit `a35a6b41`
**Status:** **DEFERRED** — refactor across T24/T25 (and possibly T17/T18); needs separate cycle
**Priority:** LOW (code-quality / DRY; no functional or security gap)
**Date filed:** 2026-04-21

---

## Context

T25 (`backend/src/api/routers/admin_audit_chain.py`) imports six helpers
across **three sibling routers** to compose its surface:

| Symbol | Origin | Purpose in T25 |
|---|---|---|
| `_audit_logs_filtered_select` | `admin.py` | Build the SQLAlchemy SELECT for the time-windowed audit-log slice |
| `_validate_audit_time_window` | `admin.py` | 422 on `until < since` |
| `require_admin` | `admin.py` | `X-Admin-Key` header gate |
| `router` | `admin.py` | Mount point for the new endpoint |
| `_operator_subject_dep` | `admin_bulk_ops.py` | Resolve `X-Operator-Subject` for `user_id_hash` audit emit |
| `_admin_role_dep`, `_admin_tenant_dep` | `admin_findings.py` (T24) | RBAC headers parser |

This produces a triangular import topology: `admin_audit_chain` → `admin`
+ `admin_bulk_ops` + `admin_findings`, each of which lives in a different
file but logically belongs to the same admin surface. The pattern is
already starting to feel like a shotgun layout — and Cycle 7 is likely
to add Webhook-DLQ (ARG-057) and emergency-throttle (ARG-052) routers
with the **same** helper-import topology.

## Problem

1. **Implicit dependency graph.** A new admin router needs to know that
   `_operator_subject_dep` lives in `admin_bulk_ops.py` (T17), not in
   `admin.py`. There is no public "admin shared kit" module to import from.
2. **Refactor risk.** Moving any one helper between sibling routers breaks
   any number of callers without a single locus of definition.
3. **Test friction.** Tests for `admin_audit_chain` end up patching
   helpers across three modules (e.g., `_resolve_chain_window` lives in
   `admin_audit_chain.py` but `_admin_tenant_dep` lives in `admin_findings.py`).

## Proposal

1. **New module** `backend/src/api/routers/admin_core.py` containing:
   * RBAC dependencies: `_admin_role_dep`, `_admin_tenant_dep`,
     `_operator_subject_dep`, `require_admin`.
   * Shared SELECT builders that multiple routers use:
     `_audit_logs_filtered_select`, `_validate_audit_time_window`.
   * The `router` `APIRouter()` instance (or a factory function).
2. **Re-export from sibling routers** to preserve backward compatibility
   with any external imports during the transition (e.g., `from .admin_core import _admin_role_dep as _admin_role_dep`).
3. **Update sibling routers** (`admin.py`, `admin_bulk_ops.py`,
   `admin_findings.py`, `admin_audit_chain.py`) to import from
   `admin_core` rather than from each other.
4. **Add a test** `tests/unit/api/test_admin_core_kit.py` asserting the
   public surface of `admin_core` (preventing accidental removal of an
   export that downstream routers depend on).

## Acceptance criteria

* (a) Cycle import graph among `admin*.py` routers reduces to a tree
  (no router imports from another router).
* (b) All existing 200/4xx behaviour preserved — sibling tests pass
  unchanged.
* (c) Public re-export shims keep external imports working
  (deprecation warning optional).
* (d) New `admin_core_kit` test pins the public surface.
* (e) Per-router file size drops by ≥10 % on average.

## Why deferred from T25

A T25-only refactor would be incomplete: the pattern emerges across at
least four routers (T17 / T18 / T24 / T25). Pulling the helpers without
also updating the siblings would create a worse intermediate state
(triangular imports + a fourth file). Bundle into a dedicated
"admin-router refactor" cycle that touches all four siblings atomically.

## Dependencies

* T17 (`admin_bulk_ops.py`), T18 (audit search/export in `admin.py`),
  T24 (`admin_findings.py`), T25 (`admin_audit_chain.py`) — all
  delivered in Cycle 6.
* Cycle 7+ admin endpoints (ARG-052 emergency-throttle,
  ARG-056 scan-schedules, ARG-057 webhook-dlq) — would benefit from
  `admin_core` being landed first.

## Rationale

KISS / DRY refactor. No functional change. Best done before Cycle 7
adds three more admin routers that would otherwise inherit the same
fragmented topology.
