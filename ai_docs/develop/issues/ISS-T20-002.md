# ISS — Bulk findings export endpoint (backend) + UI re-introduction in toolbar

**Issue ID:** ISS-T20-002
**Owner:** Backend / Admin findings + Frontend / Admin console
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — backend feature, out of scope for T20 UI fix
**Priority:** MEDIUM (operator productivity; current per-row export covers the audit-evidence path but not "snapshot of the current filtered view")
**Date filed:** 2026-04-21

---

## Context

Reviewer flagged S1-5: the original T20 toolbar surfaced a global
"Export" popover that called `downloadFindingsExport(scan_id, …)` —
but findings rendered cross-tenant and cross-scan, so the picker
silently exported one arbitrary scan. The fix cycle resolved this by
**removing the global toolbar export** and moving export into the
per-row drawer (where the operator unambiguously selects one finding
and its associated scan).

This is correct for evidence-of-a-single-finding workflows, but it
loses the operator-productivity case "export everything that matches
my current filters into a single SARIF/JUnit bundle for the
SOC inbox / SIEM ingest". That workflow needs a backend endpoint that
takes the same query string the table uses.

## Problem

* Triage analysts cannot export the current filtered view in one shot;
  they must click into each row's drawer and download N files.
* No backend route currently accepts the same filter set as
  `GET /admin/findings` and emits a SARIF / JUnit bundle.
* The toolbar slot freed by the S1-5 fix is empty UX real estate that
  operators expect to use.

## Proposal

### Backend (`backend/src/api/routers/admin_findings.py`)

1. **New endpoint** `GET /admin/findings/export`:
   * Accepts the **same** query string as `GET /admin/findings`
     (severity, confidence, false_positive, q, since, until, tenant_id, …).
   * Accepts `format=sarif|junit` (default `sarif`).
   * Streams the bundle as `application/octet-stream` with
     `Content-Disposition: attachment; filename="findings-<sha8>.<ext>"`.
   * RBAC: `super-admin` → may omit `tenant_id`; `admin` → must use
     session-bound `tenant_id` (matches the list-route invariant).
   * **Cap:** at most 5 000 findings per export (matches existing
     bulk-ops cap pattern); HTTP 413 + `detail="export_too_large"`
     above that.
   * Audit emit: `AuditEventType.ADMIN_FINDINGS_EXPORTED` with sha256
     of the canonical filter dict + count + format. NO PII / row data
     in the audit row.
   * Tests ≥ 8 cases (RBAC matrix × format × cap × empty result ×
     audit-row-emitted).

### Frontend (`Frontend/src/app/admin/findings/`)

1. **Server action** `exportAdminFindingsAction(params, format)` —
   thin wrapper around `callAdminBackendJson` (or a streaming variant)
   reusing the same `buildFindingsQuery` mapper as the list action.
2. **Toolbar button** "Экспорт страницы" (or similar) re-introduced
   next to the filter bar. Disabled when `total === 0`. Opens a
   `<select>` for SARIF / JUnit format (reuses
   `Frontend/src/components/admin/exports/ExportFormatToggle.tsx`
   if the API is compatible).
3. **Toast / progress** for the streaming download (existing
   `<DownloadProgress>` if available, or simple "Готовим экспорт…"
   spinner that resolves when the response stream ends).
4. **Tests** — Vitest ≥ 4 cases (toolbar disabled when empty;
   format selection persists in `localStorage` per session; cap
   error renders closed-taxonomy message; audit-row hint shown).

## Acceptance criteria

* (a) `GET /admin/findings/export?format=sarif` returns a valid SARIF
  2.1.0 bundle for the current filter set (schema-validated in tests).
* (b) `format=junit` returns valid JUnit XML.
* (c) Cap at 5 000 results enforced server-side; UI surfaces the cap
  via a closed-taxonomy error (`export_too_large`).
* (d) Audit row emitted on success **and** on capped failure; no PII
  in the audit row.
* (e) Frontend toolbar button is gated by RBAC (operator role does
  not see it; admin sees it tenant-scoped; super-admin sees it
  cross-tenant).
* (f) Per-row drawer export (S1-5 fix) is **not** removed — both
  flows coexist.

## Why deferred from T20 fix cycle

* T20 fix cycle was scoped to bug fixes (S0/S1/S2) on the existing
  endpoint surface. Adding a new backend route + audit event +
  schema validation is M-size on its own.
* Backend already has separate pending capacity for findings-export
  workflows (T21 bulk actions); coordinate to avoid duplicating the
  RBAC / cap / audit boilerplate.

## Dependencies

* New audit event constant `AuditEventType.ADMIN_FINDINGS_EXPORTED`
  in `backend/src/policy/audit.py`.
* Coordination with T21 (bulk findings actions) to share the
  filter-fingerprint helper.
* Possible re-use of T23's `ExportFormatToggle` UI component.

## Rationale

The per-row export covers the "single-finding evidence" case;
the bulk export covers the "current view → SIEM ingest" case.
Both are operator workflows; S1-5 was correct to remove the
ambiguous global button, but the productivity gap should not stay
open indefinitely.
