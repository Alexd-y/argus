# ISS — Promote chain markers from JSONB to dedicated `event_hash` / `prev_event_hash` columns

**Issue ID:** ISS-T25-001
**Owner:** Backend / Audit log persistence
**Source task:** T25 (Cycle 6 Batch 3 — chain integrity verification API)
**Source review:** post-review fix-cycle for commit `a35a6b41`
**Status:** **DEFERRED** — out of scope for read-only T25; requires DB migration + writer-path changes
**Priority:** MEDIUM (perf + maintainability; no security gap as long as JSONB markers stay tamper-evident)
**Date filed:** 2026-04-21

---

## Context

The current implementation of [`backend/src/policy/audit.py`](../../../backend/src/policy/audit.py) stores chain markers
(`_event_hash`, `_prev_event_hash`) inside the `details` JSONB column on
`audit_logs`. The verifier (`verify_audit_log_chain`) extracts them at read
time via `_extract_chain_markers(details)` and strips them from the canonical
projection before recomputing.

This works correctly (see T25 happy/drift/empty test coverage) but has three
ergonomic / performance drawbacks:

1. **Implicit emit-time contract.** Writers MUST remember to inject markers
   into `details` JSONB at row creation. There is no DB-level enforcement;
   a forgetful new emitter produces rows that the verifier accepts (no
   markers → falls back to fresh recompute, no drift detected). This is
   safe but reduces the "chain-of-custody" property.
2. **Slower drift forensics.** Pivoting from "chain broke at row X" to
   "what hash did writer-Y record" requires a JSONB key extraction
   (`details->>'_event_hash'`) — fine on small windows, but `verify-chain`
   on 10⁴+ events would benefit from an indexed B-tree column.
3. **Schema discoverability.** New backend engineers reading the `audit_logs`
   model see no columns named `event_hash` / `prev_event_hash`; the chain
   contract is documented only in the verifier docstring.

## Proposal

1. **Alembic migration `026_audit_log_chain_columns.py`** (additive-only, RLS-preserving):
   * Add `event_hash CHAR(64)` (NOT NULL after backfill).
   * Add `prev_event_hash CHAR(64)` (NOT NULL after backfill).
   * Backfill from `details->>'_event_hash'` / `details->>'_prev_event_hash'`
     for existing rows; rows without markers get `NULL` initially, then
     get a freshly recomputed value during a one-off backfill job (tenant by
     tenant, RLS-preserved).
   * Add B-tree index on `event_hash` for forensic pivots
     (`SELECT * FROM audit_logs WHERE event_hash = $1`).
2. **Writer path** — extend `AuditLogger.add()` to populate the new
   columns AND keep injecting JSONB markers (defence-in-depth dual-write
   for one cycle, then drop JSONB markers in a follow-up).
3. **Verifier path** — prefer the dedicated columns; fall back to JSONB
   markers when columns are NULL (transition window).
4. **Tests** — extend `tests/unit/policy/test_audit.py` and
   `tests/unit/api/test_admin_audit_chain_verify.py` to exercise both
   storage paths during the transition.
5. **Documentation** — update [`ai_docs/develop/components/audit-log-chain.md`](../components/audit-log-chain.md)
   (NEW or extend) explaining the column contract.

## Acceptance criteria

* (a) Alembic migration `026` applies cleanly upgrade + downgrade.
* (b) Backfill is idempotent (re-running produces zero diff).
* (c) Existing `verify_audit_log_chain` tests pass without modification.
* (d) New tests cover dual-storage transition (JSONB-only, columns-only,
  both populated, NEITHER populated → backfill error not chain failure).
* (e) p95 chain-verify on 10⁴ events drops from current baseline by ≥20 %.
* (f) RLS policies on `audit_logs` continue to enforce tenant isolation
  on the new columns (no cross-tenant column scan).

## Why deferred from T25

T25 is a **read-only** API addition (POST verify-chain only computes; no
writes). Adding writer-path changes + a new migration would inflate scope
beyond the cap=10 batch budget and would require coordinated rollout
with the existing audit emit producers (≥12 call sites). Promote to
Cycle 7 as a dedicated migration task.

## Dependencies

* Coordination with all `AuditLogger.add()` call sites
  (`backend/src/api/routers/*.py` + `backend/src/policy/*.py` ≈ 12 sites).
* Sustained test coverage for `tests/unit/policy/test_audit.py`.

## Rationale

JSONB markers were the right v1 choice (zero-migration cost). The dedicated
columns are the right v2 choice (DB-enforced contract + faster forensics +
better schema discoverability). Defer until next cycle to keep T25 focused
on the read-side surface.
