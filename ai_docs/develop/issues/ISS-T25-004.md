**Date filed:** 2026-04-21

## Context

T25 introduced [`_query_fingerprint`](../../../backend/src/api/routers/admin_audit_chain.py)
(a deterministic 24-hex SHA-256 of a normalized query payload) so SIEM
analytics can correlate verify-chain runs without leaking raw filter
values. The function is currently:

```python
def _query_fingerprint(
    *,
    tenant_id: str | None,
    since: datetime | None,
    until: datetime | None,
    event_type: str | None,
) -> str:
    payload = {
        "tenant_id": tenant_hash(tenant_id) if tenant_id else None,
        "since": since.isoformat() if since else None,
        "until": until.isoformat() if until else None,
        "event_type": event_type,
    }
    canonical = json.dumps(payload, sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()[:24]
```

T18 (`GET /admin/audit-logs`), T24 (`GET /admin/findings`), and T25
(`POST /admin/audit-logs/verify-chain`) all log a `query_fingerprint` for
SIEM correlation. **Each currently rolls its own helper** (small drift
risk: T25 chose 24 chars, T18 / T24 may have chosen different lengths
or different field orders, breaking cross-endpoint correlation).

## Problem

1. **Drift risk.** Three helpers, three opportunities to disagree on
   payload field order, encoding, hash truncation length. SIEM
   correlation breaks silently if T18 fingerprints a query as `abc123…`
   and T25 fingerprints the same query as `xyz789…`.
2. **No magic-number constant.** The truncation length `[:24]` appears
   inline in T25; promotion to a named `QUERY_FINGERPRINT_LEN = 24`
   constant makes the intent (24 hex chars = 12 bytes = 96-bit
   collision space, far below 2¹²⁸ but adequate for correlation, not
   identification) explicit and reviewable.
3. **Test duplication.** Each router has its own
   `test_query_fingerprint_*` cases; one shared module would consolidate
   coverage and reduce per-router overhead.

## Proposal

1. **New module** `backend/src/core/query_fingerprint.py`:
   ```python
   QUERY_FINGERPRINT_LEN: Final[int] = 24

   def query_fingerprint(
       *,
       tenant_id: str | None,
       extra: Mapping[str, Any] | None = None,
   ) -> str:
       """Stable 24-hex sha256 of a normalized query payload."""
       payload = {
           "tenant_id": tenant_hash(tenant_id) if tenant_id else None,
       }
       if extra:
           # Deep-canonicalise nested dicts; ISO-format datetimes; lowercase strings
           # to maximise correlation-without-identification.
           payload.update({k: _canonicalise_value(v) for k, v in extra.items()})
       canonical = json.dumps(payload, sort_keys=True, separators=(",", ":"))
       return hashlib.sha256(canonical.encode("utf-8")).hexdigest()[:QUERY_FINGERPRINT_LEN]
   ```
2. **Migrate T18 / T24 / T25** to import from `core.query_fingerprint`;
   delete the per-router shims.
3. **Shared tests** `tests/unit/core/test_query_fingerprint.py`:
   * Determinism (same input → same output across calls + processes).
   * Tenant-hash usage (raw tenant_id never appears in canonical bytes).
   * Field-order independence (payload `{"a":1,"b":2}` vs `{"b":2,"a":1}` → same).
   * Length invariant (`len() == QUERY_FINGERPRINT_LEN` always).
   * Datetime normalisation (`datetime` and ISO-string forms produce same fingerprint).
4. **Documentation** — new short note in
   [`ai_docs/develop/components/audit-log-chain.md`](../components/audit-log-chain.md)
   (or wherever SIEM-correlation conventions live) describing the
   fingerprint contract.

## Acceptance criteria

* (a) `core.query_fingerprint.query_fingerprint` exposed as the canonical helper.
* (b) `QUERY_FINGERPRINT_LEN = 24` named constant.
* (c) T18 / T24 / T25 routers all import from the shared module
  (zero per-router shims remain).
* (d) Same query → same fingerprint across all three endpoints
  (cross-endpoint correlation property: T25 verify-chain run on the
  same tenant + window as a T18 search produces matching fingerprints).
* (e) ≥6 unit-test cases on the shared module.
* (f) Pre-existing per-router fingerprint tests rewritten or removed
  (no duplication).

## Why deferred from T25

T25 by itself only needs *its own* fingerprint to work; promoting to a
shared helper requires touching T18 + T24 simultaneously. Bundling with
ISS-T25-003 (`admin_core.py` refactor) makes one cohesive cleanup PR.

## Dependencies

* `tenant_hash` already lives in `backend/src/core/observability.py` — reuse.
* T18 / T24 / T25 routers already deployed.
* (Optional) coordination with ISS-T25-003 admin-core refactor.

## Rationale

DRY + SIEM consistency. Cheap to do now, painful to do after Cycle 7
adds 3+ more admin endpoints each rolling their own fingerprint.
