# ISS — Migrate `findingsExport.ts` to the closed-taxonomy error pattern from `adminFindings.ts`

**Issue ID:** ISS-T20-001
**Owner:** Frontend / Findings export
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage UI)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — cross-component refactor; queue for the next cycle
**Priority:** MEDIUM (no new security gap; existing strings are already short and PII-free, but the taxonomy is implicit)
**Date filed:** 2026-04-21

---

## Context

[`Frontend/src/lib/findingsExport.ts`](../../../Frontend/src/lib/findingsExport.ts)
currently throws bare `Error` instances with hand-written messages
(`"Findings export is not available for this scan."`,
`"Findings export failed."`). Callers either pattern-match on the message
text (brittle) or display it verbatim (works but bypasses i18n).

[`Frontend/src/lib/adminFindings.ts`](../../../Frontend/src/lib/adminFindings.ts)
already encodes the canonical pattern: `AdminFindingsError` (typed `code` +
optional `status`) plus `adminFindingsErrorMessage(err)` resolving the code
to a fixed RU sentence. The shape is what the T20 review explicitly
recommended for every read/write surface in `admin/*`.

## Proposal

1. Define `FindingsExportError` mirroring `AdminFindingsError`:
   * `code: "format_unsupported" | "scan_not_found" | "forbidden" | "rate_limited" | "server_error" | "network_error"`.
   * Constructor takes `(code, status?)`.
2. Replace the two raw `throw new Error(...)` sites in
   `downloadFindingsExport()` with the typed throws (404 → `scan_not_found`;
   non-OK fallthrough → `server_error`; transport failure → `network_error`).
3. Add a `findingsExportErrorMessage(err: unknown): string` helper that
   resolves to a fixed RU sentence per code.
4. Update [`FindingsTable.tsx`](../../../Frontend/src/components/admin/findings/FindingsTable.tsx)
   and any other call sites to render `findingsExportErrorMessage(...)`
   instead of `String(err)`.
5. Backfill unit tests in `findingsExport.test.ts` covering each code.

## Acceptance criteria

* (a) `findingsExport.ts` exports `FindingsExportError` + `findingsExportErrorMessage`.
* (b) No raw `throw new Error(string)` remains in the export module.
* (c) `findingsExport.test.ts` covers each closed-taxonomy code with a happy + error case.
* (d) `FindingsTable.tsx` and any other admin export call site renders
  the localized message via the helper, never `String(err)` or `err.message`.
* (e) `npm run lint` and `npm run test:run` stay green; bundle size delta ≤ 0.5 kB gz.

## Why deferred from T20

The T20 fix-cycle scope was the cross-tenant findings TRIAGE surface
(server action, filters, focus trap, NaN-safe sort). The export module
is only touched indirectly (per-row export was moved into the drawer in
S1-5). Refactoring its error shape would inflate the patch surface and
risk breaking T23's `TenantSettingsClient` toggle plus the existing 5
green `findingsExport.test.ts` cases. The current behaviour is correct
(no PII / no stack trace leaks), only the contract is implicit.

## Dependencies

* Coordination with T23 (`TenantSettingsClient` SARIF/JUnit toggle) to
  ensure consumers don't pattern-match on the old strings.
* Optional: introduce a shared `lib/closedTaxonomyError.ts` that both
  `adminFindings.ts` and the new `findingsExport.ts` extend, to avoid
  duplicating the `code → RU string` registry per module.

## Rationale

Closed-taxonomy errors are the architectural invariant for every admin
read/write path (per the user-rules in `.cursor/rules/api-contract.mdc`).
Bringing `findingsExport.ts` in line removes a footgun for the next
operator surface that consumes the module (planned in T26 / T27).
