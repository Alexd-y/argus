# ISS — Bulk-escalate API for findings (Phase 2)

**Issue ID:** ISS-T21-001
**Owner:** Backend / Admin findings
**Source task:** T21 (Cycle 6 Batch 3 — bulk findings actions UI)
**Status:** **DEFERRED** — frontend stub shipped; backend endpoint missing
**Priority:** MEDIUM (no security gap — UI button is disabled with a tooltip; no silent no-op)
**Date filed:** 2026-04-21

---

## Context

T21 delivered the bulk-actions UI on `/admin/findings` (multi-select +
toolbar + confirm dialog). The wire today only supports
`POST /admin/findings/bulk-suppress` (used by both **Suppress** and
**Mark false-positive**), so two of the four spec'd actions are rendered
as **disabled buttons** with a tooltip pointing at this issue:

* **Escalate** — raise the severity of the selected findings (e.g. low
  → medium when SOC analysis confirms blast radius).
* **Attach to CVE** — see [ISS-T21-002](./ISS-T21-002.md).

The frontend never silently no-ops: clicking the disabled button does
nothing, the tooltip explains why, and the in-dialog Phase-2 stub also
references this issue for operators who already opened the modal.

## Proposal

Add a new endpoint `POST /admin/findings/bulk-set-severity` (or
`/admin/findings/bulk-escalate`) mirroring the contract of
`bulk-suppress`:

```http
POST /admin/findings/bulk-set-severity
Headers: X-Admin-Key, X-Admin-Role, X-Admin-Tenant, X-Operator-Subject
Body:
  {
    "tenant_id": "<UUID>",
    "finding_ids": ["<UUID>", ...],   // 1..100
    "new_severity": "critical|high|medium|low|info",
    "reason": "<str, 1..4000>"
  }
Response (2xx):
  {
    "updated_count": int,
    "skipped_no_change_count": int,
    "not_found_count": int,
    "audit_id": "<str>",
    "results": [{ "finding_id": "<UUID>", "status": "updated|skipped_no_change|not_found" }]
  }
```

RBAC: same as `bulk-suppress` (`admin` own-tenant only, `super-admin`
cross-tenant). Audit-log MUST capture old + new severity per id so
analysts can roll back via the audit-logs page (T22).

## Frontend wire-up tasks (after backend ships)

1. Extend `Frontend/src/lib/adminFindings.ts` with
   `BackendBulkSetSeverityResponseSchema` + a typed
   `bulkEscalateFindingsAction` server action (same fan-out pattern as
   `bulkSuppressFindingsAction`).
2. Flip `availability.escalate` to `true` in
   `Frontend/src/app/admin/findings/AdminFindingsClient.tsx`.
3. Render a severity dropdown inside `BulkActionDialog.tsx` for the
   `escalate` branch (currently a Phase-2 stub).
4. Add ≥4 vitest cases mirroring the suppress suite (RBAC, validation,
   partial failure, server-action error banner).

## Acceptance criteria

* (a) Backend endpoint deployed + documented in OpenAPI.
* (b) Frontend `availability.escalate` is `true` and the toolbar button
      no longer carries the Phase-2 tooltip.
* (c) `BulkActionDialog` renders a severity dropdown in the `escalate`
      branch and submits the typed payload.
* (d) New vitest cases pass; full suite stays green.
* (e) Audit-log entry captures `before_severity` + `after_severity` per
      finding id.

## Why deferred from T21

The T21 task is frontend-only by spec. Adding a new backend endpoint
would inflate the patch surface, require migrations + new permission
checks, and block the orchestration cycle on backend review. The
disabled-with-tooltip pattern keeps the UX honest (operators see the
feature exists + when it'll land) without shipping a half-done write
path.
