## Context

[`Frontend/src/app/admin/findings/AdminFindingsClient.tsx`](../../../Frontend/src/app/admin/findings/AdminFindingsClient.tsx)
keeps the URL in sync with the current filter state via:

```tsx
useEffect(() => {
  const next = filtersToSearchParams(filters).toString();
  const current = searchParams.toString();
  if (next !== current) {
    router.replace(`${pathname}?${next}`);
  }
}, [filters, /* ... */]);
```

The string-compare works correctly today but is **order-sensitive**:
`severity=critical&severity=high` ‚âÝ `severity=high&severity=critical`
even though the two filter sets are semantically identical. Every
time React re-orders an array in `filtersToSearchParams` (e.g. after
a sort tweak), the effect fires `router.replace` for a no-op
URL change, which:

* Pushes a new entry into the browser History stack (subtle UX
  noise ‚Äî Back button sees more entries than the user expects).
* Re-renders any consumer of `useSearchParams` downstream.

## Problem

* History-stack pollution; not a security or data-correctness issue
  but a paper-cut for power users.
* Future contributors might add a sort step inside
  `filtersToSearchParams` and silently double the no-op writes.

## Proposal

1. **Set-based diff** ‚Äî compare `URLSearchParams` instances by
   normalised key/value pairs instead of raw `toString()`:
   ```tsx
   function searchParamsEqual(a: URLSearchParams, b: URLSearchParams): boolean {
     const aSorted = [...a.entries()].sort();
     const bSorted = [...b.entries()].sort();
     if (aSorted.length !== bSorted.length) return false;
     return aSorted.every(([k, v], i) =>
       bSorted[i][0] === k && bSorted[i][1] === v
     );
   }
   ```
2. **Use `router.replace` only when** `!searchParamsEqual(next,
   current)`.
3. **Tests** ‚Äî Vitest:
   * filters that produce the same set in different order ‚Üí no
     `router.replace` call.
   * filters that produce a different set ‚Üí `router.replace` called
     once.
4. **Optional** ‚Äî sort the params canonically inside
   `filtersToSearchParams` (alphabetical by key, then by value)
   so the URL string is deterministic regardless of input ordering;
   then string-compare suffices.

## Acceptance criteria

* (a) `router.replace` is not called when the new filter set is
  semantically equal to the current URL.
* (b) Vitest covers ‚â• 4 ordering scenarios.
* (c) Browser History stack growth measured before/after on a
  10-keystroke-then-clear scenario shows ‚â• 80 % reduction in
  no-op entries.
* (d) URL-sharable links continue to round-trip identically
  (the canonical-sort variant is acceptable but must be
  backwards-compatible for already-shared links).

## Rationale

Set-based diff (or canonical sort + string-compare) is a small,
self-contained improvement that eliminates a class of
no-op History-stack writes. Defer until the next quality-focused
PR touches `AdminFindingsClient.tsx`.
