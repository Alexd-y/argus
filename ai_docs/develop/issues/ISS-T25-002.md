## Context

The admin audit-log endpoints — `GET /admin/audit-logs` (T18 search),
`GET /admin/audit-logs/export` (T18 export), and `POST /admin/audit-logs/verify-chain`
(T25) — share an RLS interaction pattern that is **safe but undocumented**:

* `super-admin` role with `tenant_id` omitted → cross-tenant query;
  the SQLAlchemy session is expected to operate without a
  `set_session_tenant` context (or with an explicit cross-tenant marker).
* `admin` / `operator` roles → must supply `tenant_id`; the session
  binds to that tenant and PostgreSQL RLS policies on `audit_logs`
  enforce isolation at the row level.

The pattern was **inherited from T18** (the previous cycle that exposed
the same RBAC matrix on search / export). T25 reused
`_admin_role_dep` / `_admin_tenant_dep` / `_audit_logs_filtered_select`
verbatim and therefore inherited the same RLS interaction. No new
divergence.

## Problem

1. **Undocumented operator contract.** A SOC operator running the
   verify-chain endpoint has no public document explaining when RLS
   applies vs when the super-admin path bypasses it. The contract
   lives only in router code comments + dependency-injection chain.
2. **Auditor-facing gap.** Compliance audits (SOC 2 / PCI DSS) typically
   require a written description of how privileged users access cross-tenant
   data. Currently the answer is "read the source", which is not auditor-friendly.
3. **No regression guard.** A future refactor could accidentally
   `set_session_tenant(None)` for an `admin` role (RLS bypass would silently
   succeed because the policy uses `current_setting('app.tenant_id', true)` —
   `true` returns NULL on missing setting, which makes the WHERE-clause
   match nothing rather than match everything, but still — the failure mode
   is silent).

## Proposal

1. **New doc** `ai_docs/develop/components/admin-rls-pattern.md` (~120 LoC):
   * Diagram of the request flow: header → `_admin_role_dep` → `_admin_tenant_dep` → `_enforce_chain_verify_rbac` (and siblings) → SQLAlchemy session → RLS policy.
   * Table of role × tenant_id-presence × resulting RLS behaviour
     (3 × 2 = 6 cells).
   * "What to verify" checklist for new admin endpoints
     (RBAC dep → audit emit → RLS context → integration test with
     forced cross-tenant attempt → expected denial).
2. **Cross-link** from existing per-endpoint docstrings into the new doc.
3. **Optional regression test**
   `tests/integration/api/test_admin_rls_isolation_contract.py`
   exercising the 6-cell matrix against a real PostgreSQL instance with
   RLS policies active (not the mocked unit-test path).

## Acceptance criteria

* (a) Doc explains all 3 roles × 2 tenant_id states unambiguously.
* (b) All 3 admin audit-log endpoints (search, export, verify-chain)
  cross-link to the new doc.
* (c) Optional integration test covers the 6 cells with PostgreSQL RLS
  policies actually loaded.
* (d) Compliance auditor can answer "how does ARGUS prevent
  cross-tenant audit-log access by an `admin` user" by reading the doc.

## Why deferred from T25

T25 sustained an existing pattern (T18) without changing it. The
documentation gap exists regardless of T25 and applies equally to the
two T18 endpoints already in production. A documenter pass should cover
all three siblings together rather than only T25.

## Dependencies

* Documenter agent assigned in next cycle.
* Sustained `_audit_logs_filtered_select` helper in `admin.py`.
* Sustained `_admin_role_dep` / `_admin_tenant_dep` in `admin_findings.py`.

## Rationale

Compliance posture > code-only contracts. Writing this down now (while
the pattern is fresh) is cheaper than reverse-engineering it during a
SOC 2 / PCI DSS audit. Bundle with documenter cycle, not feature cycle.
