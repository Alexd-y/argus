# ISS — Full HTTP-path performance test for chain-verify (10⁴ events ≤ 2 s p95)

**Issue ID:** ISS-T25-005
**Owner:** Backend / Performance / CI
**Source task:** T25 (Cycle 6 Batch 3 — chain integrity verification API)
**Source review:** post-review fix-cycle for commit `a35a6b41`
**Status:** **DEFERRED** — needs benchmark infra; defer to perf-CI cycle
**Priority:** MEDIUM (operational SLO; current pure-helper test gives 80 % confidence but does not measure transport overhead)
**Date filed:** 2026-04-21

---

## Context

T25's plan called for `p95 ≤ 2 s on 10⁴ events`. The shipped test suite
includes [`TestVerifyChainPerformance::test_helper_handles_10000_events_under_budget`](../../../backend/tests/unit/api/test_admin_audit_chain_verify.py)
which **measures the pure helper** (`verify_audit_log_chain`) on 10⁴
synthesised rows and asserts `< 5 s` (generous CI ceiling, production
target 2 s p95).

This is a useful tight inner-loop signal but it does **not** measure:

1. **TestClient / FastAPI request handling overhead** —
   request parsing, dependency injection (RBAC + operator + DB),
   response serialisation through Pydantic.
2. **SQL execution overhead** — the test mocks `db.execute` to return
   pre-built rows in O(1); a real PostgreSQL query against `audit_logs`
   on a 10⁴-row window adds disk + planner + RLS-policy evaluation
   latency.
3. **Structured-log emit overhead** — `logger.info` with `extra={...}`
   on every request; cumulative cost on 10⁴-events benchmarks dominated
   by the chain compute itself but worth measuring under contention.
4. **Concurrency behaviour** — what happens when 5 admins run
   verify-chain simultaneously on overlapping 90-day windows.

## Problem

* The pure-helper test gives a **lower bound** on latency. Production
  p95 will be higher; we don't know by how much.
* No CI gate fires when a regression in any of the four "outer" layers
  above silently doubles end-to-end latency.
* No baseline number for capacity planning ("how many concurrent
  verify-chain sessions can a single API pod sustain?").

## Proposal

1. **New benchmark harness** `backend/tests/perf/test_chain_verify_e2e_perf.py`
   marked `requires_postgres` + `slow`:
   * Spin up a postgres fixture, seed 10⁴ audit-log rows for a fixed
     tenant via the production `AuditLogger.add()` path (so chain
     markers are present).
   * Issue 10 sequential `POST /admin/audit-logs/verify-chain` requests
     through TestClient (warm-up 2, measure 8); record p50 / p95 / max.
   * Assert `p95 ≤ 2.0 s`; `max ≤ 5.0 s` (defensive ceiling).
   * Optional: run a 5-worker concurrent variant via `asyncio.gather`
     and assert no pathological lock contention.
2. **Performance regression gate** —
   `.github/workflows/perf-chain-verify.yml` (NEW; non-blocking
   initially / advisory like the existing T07 / T08 batch).
   Runs on a sized GH Actions runner (`large` or self-hosted) to give
   stable measurements; PR comments include the histogram.
3. **Baseline file** `backend/tests/perf/baselines/chain_verify_p95.json`
   committed to repo; CI fails with a readable diff if the new run
   regresses by >25 % vs baseline.
4. **Operator runbook**
   `ai_docs/develop/components/chain-verify-performance.md` (NEW or
   extend) documenting:
   * Reference dataset shape.
   * Expected p50 / p95.
   * Tuning knobs (`_AUDIT_CHAIN_MAX_ROWS`, `_AUDIT_CHAIN_MAX_WINDOW_DAYS`).
   * "What to do when verify-chain latency regresses" checklist.

## Acceptance criteria

* (a) `tests/perf/test_chain_verify_e2e_perf.py` exists and runs
  end-to-end against a real postgres fixture.
* (b) p95 ≤ 2 s baseline established and committed.
* (c) CI gate runs on PRs touching `backend/src/policy/audit.py` or
  `backend/src/api/routers/admin_audit_chain.py` (path-filter trigger).
* (d) Concurrent-load variant passes (5 workers × 10 requests each,
  no deadlock, p95 ≤ 3 s).
* (e) Operator runbook documents all tuning parameters.

## Why deferred from T25

* No existing perf benchmark infra in the project (`tests/perf/` does
  not exist yet); standing it up is a multi-cycle effort that would
  benefit other endpoints simultaneously (`GET /admin/findings`,
  `POST /admin/scans/bulk-cancel`, etc.).
* GH Actions runners have known noise floors that would make a 2 s SLO
  flaky without a self-hosted or `large` runner — capex coordination
  needed.
* Pure-helper test (T25 in-scope) already covers the dominant cost
  (chain compute), giving 80 % of the confidence at 5 % of the effort.

## Dependencies

* PostgreSQL fixture infrastructure (sustained from existing
  integration tests under `tests/integration/db/`).
* CI runner sizing decision (default ubuntu-latest vs `large` vs
  self-hosted).
* Coordination with capacity-planning effort (Cycle 7+).

## Rationale

End-to-end perf coverage is necessary for a production SLO commitment.
The pure-helper test is sufficient to catch order-of-magnitude
regressions today; the e2e test catches latency-budget creep that the
pure-helper test cannot see. Defer until perf-CI infrastructure is
ready cycle-wide rather than building a one-endpoint snowflake.
