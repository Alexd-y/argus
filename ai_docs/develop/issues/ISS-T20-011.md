# ISS — Strip KEV / SSVC URL params in `buildAdminFindingsUrl` when feature flag off

**Issue ID:** ISS-T20-011
**Owner:** Frontend / Admin console
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — defensive UX nit (S3-5)
**Priority:** LOW (no functional impact today; backend ignores unknown
params)
**Date filed:** 2026-04-21

---

## Context

The findings page hydrates filter state from URL search params via
`searchParamsToFilters` in
[`Frontend/src/app/admin/findings/AdminFindingsClient.tsx`](../../../Frontend/src/app/admin/findings/AdminFindingsClient.tsx)
and writes filter state back via `filtersToSearchParams`. The set of
recognised params today includes `kev` and `ssvcAction` (see
[`Frontend/src/lib/adminFindings.ts`](../../../Frontend/src/lib/adminFindings.ts)
`AdminFindingsParamsSchema`).

Two sub-features are gated by feature flags / future delivery:

* `kev_listed` — the backend response **does not currently** return a
  `kev_listed` field for every finding (only when the KEV ingest
  pipeline has run for the given CVE). The UI filter exists but
  silently no-ops on rows without the field.
* `ssvc_action` — same story; the SSVC scoring pipeline is upstream
  and not yet GA across all tenants.

If a power-user crafts a URL like
`/admin/findings?kev=true&ssvcAction=Act` while the underlying data
is sparse, the page renders an empty list with no explanation — the
filter "succeeded" but matched zero rows.

## Problem

* Confusing UX: empty page with no hint that the filter is hitting a
  data-availability gap, not a data-content gap.
* `buildAdminFindingsUrl` (and the URL writeback in
  `AdminFindingsClient`) preserves the params verbatim, so a shared
  link encodes a filter that may stop working without warning when
  the backend pipeline is reconfigured.

## Proposal

1. **Feature flag gate** — read
   `process.env.NEXT_PUBLIC_ADMIN_KEV_FILTER_ENABLED` and
   `process.env.NEXT_PUBLIC_ADMIN_SSVC_FILTER_ENABLED` (default
   `"false"`). When off:
   * `<FindingsFilterBar>` does not render the chip groups.
   * `searchParamsToFilters` strips `kev` / `ssvcAction` and emits
     a `console.warn` (dev only) noting the param was dropped.
   * `filtersToSearchParams` never writes them, so the URL
     self-heals.
2. **In-table empty-state hint** — when `total === 0` AND a gated
   filter is active in the URL despite the gate (transient race),
   render an inline help banner: "Фильтры KEV / SSVC требуют активной
   ингест-пайплайн. Уточните у super-admin."
3. **Tests** — Vitest:
   * gated off + URL has `kev=true` → filter dropped, default state
     hydrated.
   * gated on + URL has `kev=true` → filter active, URL preserved.

## Acceptance criteria

* (a) Two env vars documented in `Frontend/.env.example` with default
  `"false"` and a note about the upstream pipeline dependency.
* (b) When gates are off, the URL params for `kev` / `ssvcAction`
  round-trip to nothing (URL self-heals on first navigation).
* (c) When gates are on, behaviour unchanged.
* (d) Vitest covers both branches (≥ 4 cases).

## Why deferred from T20 fix cycle

* S3-5 grade: defensive UX nit.
* Backend KEV / SSVC pipeline GA timeline is uncertain; gating now
  is fine, but the inline help-banner copy benefits from PM input
  that the fix cycle did not solicit.

## Dependencies

* PM / docs alignment on the help-banner copy.
* Backend KEV / SSVC pipeline rollout schedule (informs default
  values of the env vars per environment).

## Rationale

Feature-flag-gating UI for upstream-pipeline-dependent features is a
sustained pattern in the project (see existing `NEXT_PUBLIC_ARGUS_*`
flags in `Frontend/.env.example`). Aligning the KEV / SSVC filters
with the same pattern removes the class of "filter exists but does
nothing" bugs.
