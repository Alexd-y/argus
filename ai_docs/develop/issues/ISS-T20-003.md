# ISS — Server-resolved tenant + subject binding via JWT/session (full implementation)

**Issue ID:** ISS-T20-003
**Owner:** Backend / Auth + Frontend / Admin console
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — auth subsystem refactor; multi-cycle effort
**Priority:** HIGH (security; current cookie-based dev shim is sufficient
for non-prod but production needs a signed token before public launch)
**Date filed:** 2026-04-21

---

## Context

T20 fix cycle introduced [`Frontend/src/services/admin/serverSession.ts`](../../../Frontend/src/services/admin/serverSession.ts)
which resolves `(role, tenantId, subject)` from cookies in the
following precedence chain:

1. `argus.admin.role` cookie (mirrored from `sessionStorage` by
   `AdminAuthContext` on mount).
2. `argus.admin.tenant` cookie (set by tenant-switcher UI).
3. `argus.admin.subject` cookie (deterministic, derived from role
   in dev / not yet wired to a real identity service in prod).
4. `NEXT_PUBLIC_ADMIN_DEV_*` env-var fallback.

This is a **transitional** shim. It is sufficient because:

* `callAdminBackendJson` always attaches the server-only `ADMIN_API_KEY`
  via `X-Admin-Key` (the actual auth credential between Next.js and the
  FastAPI backend).
* `X-Admin-Role` / `X-Admin-Tenant` / `X-Operator-Subject` are forwarded
  as **operator-attribution metadata**, not as auth claims. The backend
  RBAC dependency (`require_admin`) trusts the `X-Admin-Key` envelope,
  not the `X-Admin-Role` header.

It is **insufficient for production** because:

* Cookie-based identity is trivially forgeable in any browser dev-tool;
  if the backend ever switches from `X-Admin-Key`-only to "trust the
  role header for fine-grained RBAC", we have a privilege-escalation
  vector ready and waiting.
* The `subject` field is deterministically derived from role in the
  current shim (`admin_console:admin`, etc.), so audit rows do not
  uniquely identify the operator who performed an action.

## Problem

* No cryptographic binding between the human operator and the headers
  forwarded to the backend.
* `X-Operator-Subject` is the only field that lands in audit rows
  identifying the actor; today it is not unique per human.
* When the backend graduates from `X-Admin-Key`-only to per-operator
  RBAC enforcement, the cookie shim will need to be torn out
  atomically — risky.

## Proposal

### Option A — JWT (preferred for SaaS-native rollout)

1. **Identity provider integration** (Google Workspace / Azure AD /
   Auth0 — pick one based on customer mix). OIDC code-flow.
2. **Next.js middleware** `middleware.ts` validates the session JWT
   on every `/admin/**` request; sets `request.headers["x-admin-*"]`
   from verified claims.
3. **Backend** swaps `X-Admin-Key` for short-lived JWT bearer; OIDC
   public keys cached + rotated on a 24 h schedule.
4. **Audit subject** = `email` claim (or `sub` for service accounts).
5. **MFA enforcement** for `super-admin` role at the IdP.

### Option B — Backend session table (quicker, less SSO-friendly)

1. New `admin_sessions` table: `(session_id, role, tenant_id,
   subject, created_at, expires_at, last_used_at, ip_hash)`.
2. Backend `POST /auth/admin/login` endpoint with magic-link or
   password + TOTP; sets `argus.admin.session=<opaque_token>`
   HttpOnly Secure SameSite=Strict cookie.
3. `serverSession.ts` calls backend `/auth/admin/whoami` (server →
   server, with `ADMIN_API_KEY`) instead of trusting cookies.
4. Audit subject = the operator email stored in the session row.

### Migration plan (either option)

1. **Phase 1 (this issue)** — implement chosen flow in parallel with
   the current cookie shim, gated by `NEXT_PUBLIC_ADMIN_AUTH_MODE`
   env (`cookie` | `jwt` | `session`). Default `cookie` for dev,
   `jwt`/`session` for staging.
2. **Phase 2** — flip staging to the new mode for a week; collect
   audit-row uniqueness metrics.
3. **Phase 3** — enable in production; remove `cookie` mode + the
   `NEXT_PUBLIC_ADMIN_DEV_*` env vars from production builds.

## Acceptance criteria

* (a) `serverSession.ts` returns a `subject` that is **unique per
  operator session**, not derived from role.
* (b) Audit rows emitted by admin endpoints contain operator-unique
  subjects (verified by querying `audit_logs` after a 5-operator
  test scenario — 5 distinct subjects).
* (c) Cookie tampering in browser dev-tools no longer changes the
  `(role, tenant, subject)` triplet observed by the backend
  (verified by E2E test).
* (d) `super-admin` actions require MFA at the IdP / re-auth at the
  backend.
* (e) Operator runbook documents the new login flow + session
  lifetimes + revocation path.
* (f) Backwards-compatible: dev-loop with `NEXT_PUBLIC_ADMIN_DEV_ROLE`
  still works (gated by env mode).

## Why deferred from T20 fix cycle

* T20's reviewer flagged S0-1 (browser fetch bypassing
  `X-Admin-Key`); the fix cycle resolved that exact gap by moving
  the transport into a server action. The deeper "operator identity
  is not cryptographically bound" problem is **architectural** —
  it touches the backend `require_admin` dependency, the audit
  emit pipeline, and every admin route, not just the findings page.
* Multi-cycle scope; needs IdP procurement / capacity decisions.
* No live production exploit today (dev / staging only).

## Dependencies

* IdP procurement (if Option A) or auth-table migration (if Option B).
* Backend changes to `require_admin` (`backend/src/api/routers/admin.py`)
  to consume the new credential format.
* All admin frontend pages (`/admin/{findings,audit,tenants,scans,llm,system}`)
  inherit the change automatically once `serverSession.ts` and middleware
  are upgraded.

## Rationale

The cookie shim is honest about its scope: it documents, in code, that
identity is non-trusted UX state and that auth flows through `ADMIN_API_KEY`.
The proper fix is a real session/JWT layer; defer until SDLC bandwidth
allows a properly-staged rollout.
