## Context

After the T20 fix cycle:

* [`Frontend/src/app/admin/findings/actions.ts`](../../../Frontend/src/app/admin/findings/actions.ts) (`listAdminFindingsAction`) **ignores any
  `tenantId` parameter** for the `admin` role and uses the
  session-bound tenant from `getServerAdminSession()` instead
  (S1-6 fix). If no session tenant is bound, the action returns an
  empty page without dispatching to the backend.
* The backend `GET /admin/findings` route enforces RBAC again
  via `require_admin` ‚Äî `admin` callers cannot fetch other tenants'
  data even if the header reaches the route.

Existing regression coverage:

* `actions.test.ts` ‚Äî `"ignores admin caller's tenantId and uses
  session.tenantId instead"` (‚úÖ shipped).
* `actions.test.ts` ‚Äî `"returns an empty page (NEVER cross-tenant
  data) when admin has no session-bound tenant"` (‚úÖ shipped).
* Backend tests (`backend/tests/api/admin/test_admin_findings*.py`)
  cover the RBAC matrix server-side.

What's **missing**: a UI-level test that simulates an `admin`
operator manually crafting `?tenantId=<other-tenant-uuid>` in the URL
bar of the running browser and verifies that the rendered page does
not display cross-tenant data, even momentarily during hydration.

## Problem

* Server-action and backend tests prove the security invariant at
  the function-call boundary; an integration test would prove it
  at the user-visible boundary (URL ‚Üí DOM).
* Reviewers reading the page code can verify "looks like the action
  ignores `tenantId`" but cannot verify "after a real browser nav,
  the rendered DOM contains zero rows from tenant B".
* CI cannot catch a future regression where someone adds a
  `useEffect` that re-reads URL params and bypasses the action.

## Proposal

1. **Playwright spec** `Frontend/tests/e2e/admin-findings-rbac.spec.ts`:
   * Boots the admin console as `admin` role bound to tenant A
     (via `NEXT_PUBLIC_ADMIN_DEV_*` env or session cookie).
   * Navigates to `/admin/findings?tenantId=<TENANT_B_UUID>`.
   * Asserts: the rendered page either (a) shows the empty-state
     message ("–ê–∫–∫–∞—É–Ω—Ç –∞–¥–º–∏–Ω–∏—Å—Ç—Ä–∞—Ç–æ—Ä–∞ –Ω–µ –ø—Ä–∏–≤—è–∑–∞–Ω –∫ —Ç–µ–Ω–∞–Ω—Ç—É‚Ä¶")
     OR (b) shows tenant A's findings only ‚Äî never tenant B's.
   * Asserts: network panel records **zero** request to
     `/admin/findings` with tenant B in the URL.
   * Asserts: zero "Unhandled Runtime Error" overlay (sustained pattern).
2. **Vitest integration test**
   `Frontend/src/app/admin/findings/AdminFindingsClient.rbac.test.tsx`:
   * Mocks `listAdminFindingsAction` to spy on its arguments.
   * Mounts the page with `?tenantId=B` and an `admin` role.
   * Asserts: the action is called either (a) with no `tenantId`
     OR (b) with the session-bound tenant ‚Äî never with `B`.

## Acceptance criteria

* (a) Both tests fail when the S1-6 fix is reverted (proven by
  cherry-pick + rerun in the PR).
* (b) Both tests pass on the current main.
* (c) Playwright spec runs as part of the existing
  `admin-console.spec.ts` job (or a sibling job in the same workflow).
* (d) Documented in
  `ai_docs/develop/components/admin-rbac-test-coverage.md` (NEW or
  extend) so future contributors know the contract.

## Why deferred from T20 fix cycle

* The fix cycle's reviewer accepted the existing function-level
  coverage as sufficient for the S0/S1 closure; UI-level coverage
  is defence-in-depth, not a blocker.
* Adding a Playwright spec requires backend fixture data for two
  tenants; that fixture infrastructure is part of T27 (separate
  task in the same batch).
