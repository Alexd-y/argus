## Context

T21 delivered the bulk-actions UI on `/admin/findings` (multi-select +
toolbar + confirm dialog). The wire today only supports
`POST /admin/findings/bulk-suppress`. The **Attach to CVE** bulk
operation has no backend support, so the UI surfaces it as a disabled
button with a tooltip pointing at this issue, and the dialog stub also
references this issue.

## Proposal

Add a new endpoint `POST /admin/findings/bulk-attach-cve` (or
`/admin/findings/bulk-set-cves`) mirroring the contract of
`bulk-suppress`:

```http
POST /admin/findings/bulk-attach-cve
Headers: X-Admin-Key, X-Admin-Role, X-Admin-Tenant, X-Operator-Subject
Body:
  {
    "tenant_id": "<UUID>",
    "finding_ids": ["<UUID>", ...],   // 1..100
    "cve_ids": ["CVE-YYYY-NNNN", ...], // 1..16, regex-validated
    "merge_mode": "union|replace",     // union = add to existing, replace = overwrite
    "reason": "<str, 1..4000>"
  }
Response (2xx):
  {
    "updated_count": int,
    "skipped_no_change_count": int,
    "not_found_count": int,
    "audit_id": "<str>",
    "results": [{ "finding_id": "<UUID>", "status": "updated|skipped_no_change|not_found" }]
  }
```

Validation:

* `cve_ids` regex: `^CVE-(19|20)\d{2}-\d{4,7}$`.
* Reject CVEs that don't exist in the local CVE/intel cache (returns
  `422 validation_failed` listing the unknown ids ŌĆö never a partial
  write that would split the operator's intent across two requests).

RBAC: identical to `bulk-suppress`.

## Acceptance criteria

* (a) Backend endpoint deployed + documented in OpenAPI.
* (b) Frontend `availability.attachToCve` is `true` and the toolbar
      button no longer carries the Phase-2 tooltip.
* (c) `BulkActionDialog` renders a CVE chip input + merge-mode toggle
      in the `attach_to_cve` branch and submits the typed payload.
* (d) New vitest cases pass; full suite stays green.
* (e) Audit-log entry captures `before_cve_ids` + `after_cve_ids` per
      finding id (so analysts can roll back via the audit-logs page).
