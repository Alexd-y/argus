# ISS — `useCallback` stabilization for `onLoadMore` and similar hot props

**Issue ID:** ISS-T20-008
**Owner:** Frontend / Admin console
**Source task:** T20 (Cycle 6 Batch 3 — global findings triage console)
**Source review:** post-review fix-cycle for commit `a27c07bf`
**Status:** **DEFERRED** — performance nice-to-have (S2-3)
**Priority:** LOW (correctness unaffected; perf measured at < 5 % render
overhead today)
**Date filed:** 2026-04-21

---

## Context

[`Frontend/src/app/admin/findings/AdminFindingsClient.tsx`](../../../Frontend/src/app/admin/findings/AdminFindingsClient.tsx)
passes several callbacks down to `<FindingsTable>`:

* `onLoadMore` — invoked by the IntersectionObserver sentinel.
* `onSortChange` — invoked when the sort header is clicked.
* `onFiltersChange` — wired indirectly via `<FindingsFilterBar>`.

Today these are inline arrow functions or `useCallback`-wrapped only
where strictly required to satisfy `useEffect` dependency lists. As a
result `<FindingsTable>` receives **new function references** on every
render of the parent, defeating any `React.memo` boundary the
virtualizer or row component might benefit from.

## Problem

* The virtualised row component re-renders unnecessarily on every
  parent state change (debounced filter input, drawer open/close,
  query refetch). On large datasets this measures as 2–4 ms extra
  per scroll-tick on a mid-range laptop — well within budget but
  adds up under contention.
* If a future contributor adds `React.memo` to `<FindingsRow>`
  (see ISS-T20-006), it will silently be defeated until the
  callback identities are stabilised.

## Proposal

1. **Wrap inline handlers** in `useCallback` with the **minimum
   reasonable dependency set**:
   ```tsx
   const handleLoadMore = useCallback(() => {
     if (hasNextPage && !isFetchingNextPage) {
       fetchNextPage();
     }
   }, [hasNextPage, isFetchingNextPage, fetchNextPage]);

   const handleSortChange = useCallback((next: SortKey) => {
     setSort(next);
     // URL-sync handled by the existing effect; nothing else here
   }, []);
   ```
2. **Add `React.memo`** to `<FindingsRow>` (after ISS-T20-006
   decomposition lands) with a custom comparator that ignores
   prop-identity churn on the row item (compares `id` + `updated_at`).
3. **Profile** before/after with React DevTools Profiler on the
   1 000-row Vitest fixture (already exists in
   `FindingsTable.test.tsx`); document the delta in the PR
   description.
4. **Tests** — assert via `useRef` spy that
   `onLoadMore`/`onSortChange` identities do not change across
   parent renders that don't touch their dependencies.

## Acceptance criteria

* (a) `onLoadMore`, `onSortChange`, `onClose` (drawer) all wrapped
  in `useCallback`.
* (b) `<FindingsRow>` (post ISS-T20-006) memoised; comparator
  documented inline.
* (c) Profiler delta ≥ 30 % reduction in `FindingsTable` render
  count for the "open / close drawer" scenario on the 1 000-row
  fixture.
* (d) No new Vitest failures; existing 18 cases remain green.
* (e) No new `react-hooks/exhaustive-deps` warnings.

## Why deferred from T20 fix cycle

* S2-3 grade: no functional or security impact.
* Best paired with ISS-T20-006 (component decomposition) so that
  `React.memo` actually has a stable boundary to wrap. Doing it
  before the decomposition would just churn the same 700-line file.

## Dependencies

* ISS-T20-006 (decomposition) lands first or in the same PR.

## Rationale

Stable callback identities + memoisation is the standard pattern for
list-virtualised UIs; it's small, well-understood, and pairs
naturally with the planned decomposition. Defer until both can land
together as a single perf-focused PR.
