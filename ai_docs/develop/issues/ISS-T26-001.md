# ISS — Accent-on-dark contrast for primary admin buttons (WCAG 2.1 AA)

**Issue ID:** ISS-T26-001
**Owner:** Frontend / Design system
**Source task:** T26 (Cycle 6 Batch 6 — axe-core a11y gate for admin pages)
**Status:** **DEFERRED** — flagged by axe-core gate; needs design-system decision
**Priority:** MEDIUM (real WCAG AA failure on primary action; affects ≥7 admin buttons; low likelihood of operator harm because the buttons are still discoverable, but external accessibility review will reject the build)
**Date filed:** 2026-04-21
**WCAG ref:** 1.4.3 Contrast (Minimum) — AA
**Axe rule:** [`color-contrast`](https://dequeuniversity.com/rules/axe/4.11/color-contrast)

---

## Context

The new T26 axe-core gate caught one persistent a11y failure on the
`audit logs: chain-verify success banner` interactive scenario. The
violation is on the **primary admin action button** styled with the
brand purple fill + dark text:

```html
<button
  class="… border-[var(--accent)] bg-[var(--accent)] … text-[var(--bg-primary)] …"
  data-testid="audit-verify-chain"
>
  Verify chain integrity
</button>
```

* Foreground (`var(--bg-primary)`): `#0a0a0a` — axe sees `#0b0b0b`
  (sub-pixel rasterization)
* Background (`var(--accent)`): `#A655F7` — axe sees `#974ee0`
  (composited over `--bg-secondary`)
* **Measured contrast: 4.20 : 1**
* WCAG AA threshold for normal text (14 px / 10.5 pt): **4.5 : 1** → fails

The same fill + dark-text combination is used by other primary actions
across the admin console — see the “Affected surfaces” section. We do
not patch a single component because the design system token
(`--accent` + dark text) is reused, and a one-off override would drift
from brand consistency.

## Why this is not a 1-line component fix

Three otherwise-tempting paths each have a real downside:

1. **Local `text-black` override** — bumps contrast from 4.20 → ~4.62 :1
   on the *current* axe reading, but is brittle: any future change to
   `--bg-primary` re-introduces the failure, and we end up with a
   single button that lies about its design-token usage.
2. **Switch the accent to `--accent-dim` (`#8b44d4`)** — gives 5.05 :1
   with `text-white`, but flips the visual hierarchy: dim is currently
   reserved for hover/pressed/secondary states. Re-defining "primary"
   without team review is a brand decision, not a typo.
3. **Bump `--accent` lightness globally** — improves contrast across
   the seven buttons in one shot, but the accent is the brand color
   and is referenced by borders, focus rings, glow effects, and the
   marketing surface; a global lightness change needs design sign-off.

Any of these is a ≤ 5-line change once the trade-off is settled. The
T26 task spec explicitly told the implementer to defer non-trivial
fixes to a tracked issue, so we file this one and gate the failing
scenario with `test.fail` (see Frontend/tests/e2e/admin-axe.spec.ts).

## Proposal

Pick one of the following in a follow-up Cycle:

* **Option A (preferred — minimal blast radius):** introduce a new
  token `--accent-strong: <darker purple, L ≤ 0.16>` plus
  `--on-accent: <off-white>`. Migrate all primary action buttons to
  `bg-[var(--accent-strong)] text-[var(--on-accent)]`. Keep `--accent`
  as the brand color for borders, glows, focus rings, and decorative
  surfaces.
* **Option B:** redefine `--accent` itself to a darker/stronger shade
  (e.g. `#8B44D4` — currently `--accent-dim`) and rename the existing
  brighter tone to `--accent-glow` for decorative usage.
* **Option C (escape hatch, NOT recommended):** keep the bright accent
  but bump button text to bold + 16 px so axe drops the threshold to
  3:1 ("large text"). Rejected because it inflates button height
  across the admin console and breaks layout density.

## Acceptance criteria

* (a) All 7 primary admin buttons listed above pass axe `color-contrast`
      against WCAG 2 AA (≥ 4.5 : 1).
* (b) Visual diff against current Storybook / Chromatic snapshots is
      reviewed and explicitly approved by design.
* (c) `Frontend/tests/e2e/admin-axe.spec.ts` removes the `test.fail`
      annotation on `audit logs: chain-verify success banner` and the
      assertion turns into a normal `toEqual([])`.
* (d) New tokens (if Option A) are documented in
      `ai_docs/architecture/design-tokens.md` (or wherever the system
      lives — currently the tokens are inline in
      `Frontend/src/app/globals.css`).
