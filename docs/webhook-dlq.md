# Очередь мёртвых вебхуков (webhook DLQ) — runbook для операторов

> **Статус**: Cycle 6 Batch 5, ARG-053. Аудитория: администраторы платформы и on-call.  
> **Язык**: краткое операторское руководство; детали реализации — в коде и миграциях.

Документ описывает, **где** хранятся записи о неуспешных доставках уведомлений, **как** ими пользоваться через API и UI, **как** фоновый планировщик переотправляет попытки и **кто** к чему имеет доступ. Секреты, токены и полные тела ошибок **не** фиксируйте в тикетах публично — опирайтесь на идентификаторы записей и структурированные логи.

---

## 1. Модель данных

Персистентное хранилище введено миграцией **Alembic 027** (`backend/alembic/versions/027_webhook_dlq.py`).

| Объект | Назначение |
| ------ | ---------- |
| Таблица `webhook_dlq_entries` | Очередь «мертвых» доставок вебхуков: что не удалось доставить после встроенных ретраев в адаптере, с полем для планирования следующей автоматической попытки. |

Среди колонок: привязка к тенанту, тип адаптера, идентификаторы события, хеш целевого URL (не сам URL), сериализованный payload для повторной отправки, код последней ошибки, счётчик попыток, моменты следующей попытки / успешного replay / ручного или автоматического отказа. На PostgreSQL на таблицу действует RLS в духе остального ядра (тенант из сессии).

## 2. Админ-API

Префикс (от корня маршрутов backend): **`/admin/webhooks/dlq`**. Админ-маршруты DLQ смонтированы именно здесь, **не** под `/api/v1/admin/...`, чтобы снизить путаницу у операторов.

| Метод и путь (шаблон) | Назначение |
| --------------------- | ---------- |
| `GET /admin/webhooks/dlq` | Список записей с фильтрами и пагинацией (тенант задаётся контекстом вызова; см. RBAC). |
| `POST /admin/webhooks/dlq/{entry_id}/replay` | Повторная отправка: собирается адаптер **заново**, как при админ-операции, без наследования служебного состояния диспетчера. |
| `POST /admin/webhooks/dlq/{entry_id}/abandon` | Явная пометка записи как отброшенной оператором (с записью в аудит). |

Ответы и коды ошибок укладываются в **согласованный перечень** коротких `detail` (без вольных длинных сообщений) — на фронтенде отображаются человеко-понятные пояснения, а не внутренние трейсбеки. При обращении к чужому тенанту спец-статус «запрещено» **не** выдаётся — запись ведёт себя как **не найденная**, чтобы не подтверждать существование ID.

Реализация: `backend/src/api/routers/admin_webhook_dlq.py`. Клиентскую часть API удобно смотреть в `Frontend/src/lib/adminWebhookDlq.ts` (типы и сопоставление `detail` с действиями на UI); серверные actions страницы — `Frontend/src/app/admin/webhooks/dlq/actions.ts`.

## 3. Celery Beat: авто-replay и отказ по возрасту

Периодическая задача: **`argus.notifications.webhook_dlq_replay`** (зарегистрирована в `backend/src/celery/beat_schedule.py`).

- **Расписание:** ежедневно в **06:00 UTC** (отдельная очередь `argus.notifications`, чтобы не мешать сканам и отчётам).
- **Replay:** выбираются записи, у которых наступил `next_retry_at`, батчами; при неуспехе увеличивается `attempt_count` и пересчитывается следующая попытка.
- **Экспоненциальный backoff:** задержка между попытками **30 с × 4^N**, **потолок 24 h**, **без джиттера** (согласовано с расчётом в сервисе уведомлений).
- **Авто-отказ:** записи старше **14 суток** (константа `DLQ_MAX_AGE_DAYS`) без успешного replay помечаются отброшенными с причиной `max_age`. Ручной **abandon** оператором — отдельный путь, фиксируется в аудите.

## 4. Веб-интерфейс

Раздел: **`/admin/webhooks/dlq`** (маршрут Next.js: `Frontend/src/app/admin/webhooks/dlq/`).

- Таблица записей, фильтрация, диалоги **replay** и **abandon**, отображение кодов ошибок в пользовательском виде.  
- Не вставляйте в саппорт-тикеты **сырые** payload из БД; при разборе инцидентов хватит id записи, тенанта и времени.

## 5. RBAC: admin и super-admin

- **Operator** (роль в админ-конвейере проекта) к этим endpoint’ам **не** допускается: ответ **403** с нейтральным `detail` (см. заголовок модуля в `admin_webhook_dlq.py`).
- **admin** — видит и действует **только в рамках своего тенанта** (нужен корректный контекст тенанта; несоответствие — как отсутствие записи).
- **super-admin** — может работать **межвенантно** (в т.ч. список с учётом опционального фильтра тенанта); правила **не** раскрывают чужие записи через отличающийся код ответа.

## 6. Чеклист on-call (коротко)

1. Миграция **027** применена, Celery **beat** и **worker** для очереди `argus.notifications` запущены.  
2. Симптом «уведомления не уходят» сначала проверяйте интеграции и health адаптеров; затем раздел **DLQ** — есть ли записи в статусе ожидания.  
3. **Replay** — если уверены в текущем канале/учётке; **abandon** — если доставка больше не нужна.  
4. Массовый застой по времени: смотрите график задачи `webhook_dlq_replay` и логи с полем `argus_event` (например, `webhook_dlq_replay_tick`), **не** прикладывайте стектрейсы к пользовательским чатам.
