## 1. Бэклог (что сознательно не доведено)

### Из **v5** (`argus_implementation_prompt_v5.md` + отчёт `2026-04-03-argus-v5-orchestration-report.md`)

- **120+ новых MCP tools** под REST, которых в бэкенде ещё нет: pause/resume/retry scan, compare/clone/export scan, расширенный finding/report management, recon engagements, threat modeling, VA/exploitation по engagement, расширенный intelligence (CVE search, exploitdb, WAF/CDN, CT logs, whois, geoip и т.д.).
- Доведение до **150+ явных** `@mcp.tool()` (не только Kali registry) и полный **`test_mcp_tools.py`** по перечню v5 — после паритета API.
- **Блок 6 v5** (частично): timeline, statistics, false-positive POST, remediation GET и др. — если не покрыты отдельными задачами.
- **Усиление prod:** при пустом `ADMIN_API_KEY` cache-роутер может оказаться фактически открытым; согласовать политику (обязательный ключ или deny-by-default).

### Из **v4 / HexStrike-сравнения** (ранее в планах)

- Полный **TOOL_MAP** с парсерами вывода CLI, загрузка артефактов в MinIO для каждого инструмента.
- **~150 сырых CLI** как у HexStrike — не цель текущего ARGUS, но в бэклоге как продуктовый паритет.
- **ScanKnowledgeBase** в v5 уже сделан; дальше можно: глубже связать с отчётами, планировщиком сканов, UI.

### Из **документации**

- **Follow-up scan API** для Valhalla (`POST .../follow-up`) — в `docs/reporting.md` как будущая опция, сейчас ручной новый скан.
- **Аутентификация клиента к ARGUS API** (Bearer и т.д.) — в контракте фронта помечена как не реализованная.
- **Kubernetes / полноценный prod-hardening** — в архитектурных docs часто «out of scope, Compose».

### Операционное

- Полный **`pytest tests/`**, CI, canary — из отчётов Strix/v5 как рекомендации, не как «дыра в коде».

## 2. Заглушки и слабые места в коде

### Устаревшие **тексты** (поведение уже может быть нормальным)

- В **`mcp-server/argus_mcp.py`** у `get_scan_memory_summary`, `get_process_list`, `kill_process` в docstring всё ещё написано *«stub 501 until implemented»*, хотя в **API роутерах `501` по grep не используется** — имеет смысл **обновить описания** под фактические ответы (200/403/404/JSON с ошибкой).

### Роутеры / сервисы

- **`api/routers/tools.py`** — в шапке и комментариях: *«MVP stubs»* / стubs → `execute_command`; логика может быть рабочей, но уровень зрелости задокументирован как MVP. Везде убрать MVP, т.к. это уже saas решение
- **`core/auth.py`** — валидация API key: комментарий про **stub**, доработка в «Phase 3+».
- **`api/routers/auth.py`** — `user_id = "stub-user-id"` для части сценариев.
- **`orchestration/exploit_verify.py`** — **`_run_poc_stub`**, TODO **ARGUS-008**: реальный безопасный runner PoC вместо заглушки.
- **`api/schemas.py`** — в описании поля про **recovery** может остаться формулировка «stub until recovery»; recovery уже внедрён — стоит **подчистить описание**.

### Recon / адаптеры

- **`recon/adapters/security/*`** — явные **stub**: `prowler_adapter`, `checkov_adapter`, `terrascan_adapter`, `scoutsuite_adapter`, `trufflehog_adapter`.
- **`data_sources/*`** — **stub-клиенты**: HIBP, SecurityTrails, VirusTotal, Censys (заголовки файлов).
- **`recon/va_http_audit.py`** — **HTTP stub** для артефактов (намеренно не полный реплей запросов).
- **`recon/step_registry.py`** — `STUB_STEPS` сейчас пустой; механизм «stub steps» в пайплайне есть.

### Отчёты / шаблоны

- **`services/reporting.py`**, **`jinja_minimal_context.py`** — **`tier_stubs`** / видимость секций по тиру: это **не API-заглушка**, а шаблонная логика «заглушки секций» в отчёте.

### MCP

- **`mcp-server/main.py`** — комментарий *«stub for Phase 1»* (может относиться к entrypoint, не ко всему серверу).
