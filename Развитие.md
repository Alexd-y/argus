Я проверил публичные материалы: Project Glasswing от Anthropic позиционируется как закрытая инициатива для защиты критического ПО с Claude Mythos Preview, доступом для крупных партнёров и упором на поиск сложных уязвимостей в first-party и open-source коде. По сообщениям СМИ, вокруг Mythos уже есть отдельный риск-класс: контроль доступа, misuse, governance и безопасность самого AI-инструмента. Источники: [Anthropic Glasswing](https://www.anthropic.com/glasswing), [Anthropic project page](https://www.anthropic.com/project/glasswing), [Guardian](https://www.theguardian.com/technology/2026/apr/22/anthropic-investigates-report-of-rogue-access-to-hack-enabling-mythos-ai), [TechRadar](https://www.techradar.com/pro/security/mythos-accessed-by-unauthorized-users-as-anthropic-says-were-investigating-cracks-may-be-showing-in-project-glasswing-as-unknown-users-access-model-via-third-parties).

Главная мысль: ARGUS не должен пытаться быть “ещё одним сканером” или “ещё одним AI-пентестером”. Чтобы конкурировать с Glasswing/Mythos-классом, ARGUS должен стать evidence-native security operating system: платформа, где каждое утверждение связано с артефактом, каждое AI-решение проверяется, каждый риск воспроизводим, а remediation доходит до кода, CI/CD и владельца системы.

**Стратегическая цель**

ARGUS должен стать платформой для доказательной защиты критического ПО:

- не “мы нашли уязвимость”, а “мы доказали, воспроизвели, оценили риск, предложили патч, проверили фикс и сохранили цепочку доказательств”;
- не “AI написал отчёт”, а “AI помог, но quality gate, evidence graph и валидаторы не дали ему соврать”;
- не “сканирование сайта”, а continuous security assurance для web, API, source code, cloud, containers, dependencies, identity, supply chain и AI-систем.

**1. Evidence Graph как ядро ARGUS**

Нужно сделать центральную модель данных:

- `Asset`: домен, сервис, endpoint, repo, container image, cloud resource.
- `Artifact`: raw output tool, HTTP response, TLS cert, code slice, SBOM, log, screenshot.
- `Evidence`: нормализованный факт с source, timestamp, confidence, hash.
- `Finding`: только вывод из evidence, не из AI-текста.
- `Validation`: reproduced / unverified / disproven / inconclusive.
- `Control`: OWASP, WSTG, ASVS, NIST, CIS, ISO, MITRE ATT&CK.
- `Remediation`: owner, patch, test, verification result.

Это даст ARGUS главное конкурентное отличие: доказательность. Mythos может быть сильнее как модель, но enterprise-клиентам нужна не магия, а воспроизводимый audit trail.

**2. Agentic Security Engine, но с жёсткими границами**

ARGUS должен иметь несколько специализированных агентов:

- Recon Agent: собирает активы, DNS, TLS, headers, ports, URLs.
- Web/API Agent: тестирует endpoints, auth, rate limiting, access control.
- Code Audit Agent: читает репозиторий, строит data-flow, ищет vuln patterns.
- SCA/SBOM Agent: зависимости, CVE, EPSS, exploit maturity, reachability.
- Cloud/IaC Agent: Terraform, Kubernetes, IAM, secrets, public exposure.
- Exploitability Verification Agent: подтверждает только безопасными, разрешёнными PoC.
- Remediation Agent: предлагает patch, test, config change.
- Report QA Agent: запрещает unsupported certainty.

Но важное правило: autonomous exploitation должен быть policy-bound. Никакого “автономного взлома” без scope, rate limits, authorization, safety policy и kill switch.

**3. Класс Mythos-конкуренции: code-first vulnerability discovery**

Чтобы ARGUS реально приблизился к Glasswing-классу, нужен серьёзный code analysis слой:

- interprocedural taint analysis;
- path-sensitive security reasoning;
- framework-aware rules для Django, FastAPI, Express, Next.js, Spring, Laravel, Rails;
- authz graph: кто может вызвать endpoint, какие роли нужны, где broken access control;
- secret flow: от env/secrets до logs/responses;
- unsafe deserialization, SSRF, injection, IDOR, file upload, template injection;
- reachability-aware SCA: не просто “CVE в package-lock”, а используется ли vulnerable path;
- fuzz harness generation;
- patch generation + regression security tests.

Это должно стать отдельным продуктовым направлением: ARGUS Critical Code Review.

**4. Continuous Security Assurance вместо разовых отчётов**

Мировой лидер не продаёт “PDF после скана”. Он продаёт постоянную уверенность.

Нужно добавить:

- baseline security posture per asset;
- scheduled scans;
- delta reports: что изменилось после deploy;
- CI/CD gates;
- PR security review;
- “fix verified” workflow;
- SLA tracking;
- executive risk trends;
- asset owner mapping;
- Jira/Linear/GitHub issue sync.

Отчёт должен стать не конечным продуктом, а снимком состояния evidence graph.

**5. Report Quality как бренд ARGUS**

Недавние правки отчёта были правильным направлением. Это нужно развить в отдельную систему доверия:

- `No evidence -> no claim`.
- `Tool failed -> not assessed`.
- `Weak evidence -> unverified`.
- `AI text -> linted, validated, traceable`.
- CVSS consistency enforcement.
- Duplicate finding merge.
- Business risk proportionality.
- No fake exploit chains.
- No “comprehensive pentest” при quick scan.

Это можно прямо позиционировать как ARGUS Truth Layer.

**6. Enterprise-grade governance**

Если ARGUS хочет работать с критическим ПО, нужны функции уровня банка/гос/defense:

- on-prem / air-gapped deployment;
- full audit logging;
- signed artifacts;
- evidence hashing;
- model prompt/version tracking;
- RBAC/ABAC;
- tenant isolation;
- data retention policy;
- encrypted artifact vault;
- approval workflows для active testing;
- legal scope enforcement;
- model output safety filters;
- no training on customer data by default.

После историй вокруг несанкционированного доступа к Mythos это станет конкурентным преимуществом: ARGUS должен быть не только powerful, но и controllable.

**7. Поддержка AI-era угроз**

ARGUS должен проверять не только классические web/appsec проблемы, но и AI-системы:

- prompt injection;
- tool injection;
- data exfiltration through agents;
- insecure RAG;
- over-permissive tools;
- model supply chain;
- MCP/tool permission abuse;
- agent sandbox escape;
- insecure eval pipelines;
- AI-generated code vulnerability drift;
- policy bypass;
- sensitive data leakage in prompts/logs.

Это направление надо оформить как ARGUS AI Security Module.

**8. Партнёрская стратегия**

Anthropic выигрывает партнёрствами. ARGUS тоже нужен ecosystem play:

- интеграции с GitHub, GitLab, Azure DevOps;
- cloud integrations: AWS, GCP, Azure;
- SIEM/SOAR: Splunk, Sentinel, Chronicle;
- ticketing: Jira, Linear;
- scanners: Nuclei, Semgrep, CodeQL, Trivy, Grype, Syft, ZAP, Burp, Nmap;
- compliance: SOC 2, ISO 27001, PCI DSS, HIPAA, NIST SSDF, OWASP ASVS;
- open-source maintainers program;
- bug bounty / responsible disclosure workflow.

Нельзя всё писать самим. ARGUS должен быть orchestration + intelligence + evidence layer поверх лучших инструментов.

**Ключевой продуктовый тезис**

ARGUS должен конкурировать не обещанием “наш AI найдёт zero-days”, а более сильным обещанием:

“ARGUS continuously proves what is secure, what is vulnerable, what was not assessed, how we know it, how to fix it, and whether the fix worked.”

Это честнее, полезнее для enterprise и лучше защищает от главной проблемы AI security tooling: уверенного, но недоказанного вывода.
